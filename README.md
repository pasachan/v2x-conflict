# v2x-conflict-5g

At what vehicle density does **intent conflict** occur in a 5G software-defined
vehicle (SDV) network, and can QoS-protected scheduling move that point?

This is a self-contained ns-3 / 5G-LENA simulation study of a single 5G NR
mid-band cell serving vehicles that each request a network service ("intent")
with its own service-level agreement (SLA). As the number of vehicles grows,
the cell eventually cannot satisfy every intent's SLA at once. We define the
**Conflict Density (CD)** as the minimum vehicle density at which at least one
intent violates its SLA, and estimate it with confidence intervals over many
random seeds.

## Requirements

- [ns-3.45](https://www.nsnam.org/) with the
  [5G-LENA NR module](https://5g-lena.cttc.es/) (nr v4.1) built and working.
- Python 3 with `numpy`, `pandas`, `scipy`, `matplotlib` (for the analysis).

## Layout

```
main.cc                      command-line entry point
config/ran-config.{h,cc}     RAN bring-up (spectrum, channel, antennas, MIMO, CSI)
config/fiveg-config.{h,cc}   5G mid-band numeric parameters
traffic/intents.{h,cc}       intent catalogue + SLAs
scenario/scenario.h          run parameters
scenario/scenario-runner.cc  topology, mobility, traffic, bearers, FlowMonitor
scenario/single-intent.cc    Case 1 assignment (all remote driving)
scenario/multi-intent.cc     Case 2 assignment (round-robin mix)
metrics/sla-monitor.{h,cc}   per-intent SLA evaluation + CSV output
validation/runner.py         run engine (one binary invocation per point, resumable)
validation/run_study.py      full multi-seed density grid
validation/analyze.py        statistics, figures, LaTeX tables
```

## Installing into ns-3

Copy this folder into the ns-3 `scratch/` directory:

```
ns-3.45/scratch/v2x-conflict-5g/
```

It has its own `CMakeLists.txt`, so ns-3 builds it as the target
`v2x-conflict-5g`.

```bash
cd ns-3.45
./ns3 build
```

## Running

Single run:

```bash
./ns3 run "v2x-conflict-5g --scenario=single --numVehicles=14"
./ns3 run "v2x-conflict-5g --scenario=multi  --numVehicles=30 --qosProtected=1"
```

Key options: `--scenario` (`single`/`multi`), `--numVehicles`, `--simTime`,
`--speed` (m/s, default 13.89 = 50 km/h), `--qosProtected`, `--rngRun`
(seed / independent channel realisation), `--csvPath`.

Full conflict-density study (multi-seed density grid, resumable) and analysis:

```bash
python3 scratch/v2x-conflict-5g/validation/run_study.py --workers 8
python3 scratch/v2x-conflict-5g/validation/analyze.py
```

Results (CSV) land in `results/`; figures and LaTeX tables in
`results/analysis/`. Both are git-ignored and regenerated by the scripts.

## Scenario

One gNB (25 m mast) stands 10 m from a straight 500 m road. `N` vehicles are
spaced evenly along the road and drive at 50 km/h. The channel is the 3GPP
Urban Macro model. Each vehicle runs one downlink UDP constant-bit-rate flow
on a dedicated EPS bearer whose 5QI matches its intent; throughput, mean
one-way delay, and loss are measured per flow with FlowMonitor.

Radio configuration (5G mid-band macro cell): 3.5 GHz carrier, 100 MHz
bandwidth, 30 kHz subcarrier spacing (numerology 1), 8x8 gNB antenna array,
rank-4 MIMO with CSI-RS link adaptation, OFDMA proportional-fair scheduler.

### Intents

| Intent | 5QI | SLA latency | Rate |
|---|---|---|---|
| remote-driving | 85 | 5 ms | 20 Mbps |
| cooperative-awareness | 79 | 50 ms | 0.5 Mbps |
| sensor-sharing | 83 | 50 ms | 10 Mbps |
| infotainment | 2 | 150 ms | 5 Mbps |
| hd-map-update | 79 | 50 ms | 5 Mbps |

Two demand cases: **single** (every vehicle demands remote driving) and
**multi** (the five intents assigned round robin, 20% each).

A flow violates its SLA if throughput < 95% of its rate, mean delay exceeds
its budget, or loss exceeds its limit. A density is in conflict if at least
one flow violates (the analysis also reports stricter 5/10/25% criteria).

### QoS-protected scheduling

With `--qosProtected`, the cell uses the OFDMA QoS scheduler and gives remote
driving and sensor sharing guaranteed-bit-rate (GBR) reservations; the other
intents share the leftover capacity ("remote driving gets dedicated
resources, sensor sharing second, the rest share what is left").

## What the study produces

- conflict density per configuration with 95% confidence intervals (10-seed
  style multi-seed estimation; the shipped grid uses 8 seeds);
- a conflict-probability curve P(conflict | N) with Wilson intervals;
- a first-failure analysis (which intent and which SLA term breaks first);
- a conflict-criterion sensitivity analysis;
- the effect of QoS-protected scheduling on all of the above.

## Modelling notes

These settings are load-bearing; changing them casually breaks the model:

- **CSI-RS CQI feedback** (`CsiFeedbackFlags=2`) with a quasi-omni gNB beam.
  The nr default (PDSCH-based CQI) destabilises link adaptation with several
  UEs per cell.
- **Isotropic antenna elements**: vehicles pass the gNB on both sides, so a
  single directional sector would put half the road behind the array.
- **RLC UM, 1 MB buffer**: large enough that congestion shows as queueing
  delay, small enough to avoid a known 5G-LENA overload abort that only
  occurs far past the conflict point (the study counts such a crash as a
  conflict).
- The CD is a per-cell figure under a strict worst-vehicle criterion, so it
  should be read as a conservative lower bound; a sectorised, multi-cell
  deployment supports more.
